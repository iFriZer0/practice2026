# RS-485 Microservice

gRPC-микросервис для взаимодействия с драйвером RS-485.

> **Проект предназначен для сборки и запуска под Linux (Ubuntu 24.04 или совместимые дистрибутивы).**

## Содержание

* Структура проекта
* Требования
* Рабочая сборка
* Очистка рабочей сборки
* Запуск микросервиса
* Тестирование
* Очистка тестовой сборки
* Генерация gRPC-кода

---

# Структура проекта

```text
rs/
├── api/                 # .proto-файлы драйвера и микросервиса
├── client/              # gRPC-клиенты драйвера и микросервиса
├── common/              # Общие типы, обработка ошибок и вспомогательные функции
├── config/              # Конфигурационный файл и загрузчик конфигурации
├── fake_driver/         # Тестовая реализация драйвера RS-485
├── service/             # Реализация микросервиса
├── test_client/         # Консольный клиент для тестирования
├── scripts/             # Скрипты сборки, запуска, очистки и генерации
├── build/               # Рабочая сборка (создаётся автоматически)
├── build-test/          # Тестовая сборка (создаётся автоматически)
├── CMakeLists.txt
└── README.md
```

# Требования

Для сборки проекта необходимы:

* Linux (Ubuntu 24.04 или совместимый дистрибутив);
* CMake 3.16 или новее;
* компилятор с поддержкой C++17;
* Protobuf;
* gRPC;
* yaml-cpp;
* pkg-config;
* GoogleTest — только для автоматических тестов.

Установка зависимостей:

```bash
sudo apt update

sudo apt install \
    build-essential \
    cmake \
    pkg-config \
    protobuf-compiler \
    libprotobuf-dev \
    libgrpc++-dev \
    protobuf-compiler-grpc \
    libyaml-cpp-dev \
    libgtest-dev
```

Проверить установку:

```bash
cmake --version
protoc --version
pkg-config --modversion grpc++
```

# Рабочая сборка

Все команды выполняются из директории `rs`.

```bash
cd rs
./scripts/build.sh
```

Скрипт:

* создаёт каталог `build`;
* выполняет конфигурацию CMake;
* генерирует C++- и gRPC-код из `.proto`-файлов;
* собирает исполняемый файл `rs485_microservice`.

После успешной сборки бинарный файл находится по пути:

```text
build/rs485_microservice
```

# Очистка рабочей сборки

Для удаления каталога `build` выполните:

```bash
./scripts/clean.sh
```

После очистки рабочую версию необходимо собрать заново:

```bash
./scripts/build.sh
```

# Запуск микросервиса

Перед запуском микросервиса должен быть доступен реальный драйвер RS-485 либо тестовый `rs485_fake_driver`.

Рабочую версию рекомендуется запускать с помощью скрипта:

```bash
./scripts/run.sh
```

Скрипт:

* проверяет наличие `build/rs485_microservice`;
* проверяет наличие конфигурационного файла;
* устанавливает переменную окружения `RS485_CONFIG_PATH`;
* запускает микросервис.

По умолчанию используется конфигурационный файл:

```text
config/config.yaml
```

Другой конфигурационный файл можно передать первым аргументом:

```bash
./scripts/run.sh /полный/путь/config.yaml
```

Для остановки микросервиса нажмите:

```text
Ctrl + C
```

# Тестирование

Проект поддерживает три варианта тестирования:

1. через общий графический интерфейс;
2. вручную через консольный тестовый клиент;
3. автоматически с помощью GoogleTest и CTest.

Во всех вариантах тестовая сборка использует:

* `rs485_fake_driver` — тестовую реализацию драйвера;
* `rs485_microservice` — тестовый экземпляр микросервиса;
* адреса из `config/config.yaml`.

## Сборка тестовой версии

Выполните:

```bash
./scripts/build_tests.sh
```

Скрипт:

* создаёт каталог `build-test`;
* включает CMake-опцию `RS485_BUILD_TESTING=ON`;
* генерирует C++- и gRPC-код;
* собирает программы для ручного и автоматического тестирования;
* не запускает тесты автоматически.

После успешной сборки создаются:

```text
build-test/
├── rs485_microservice
├── rs485_fake_driver
├── rs485_test_client
└── rs485_integration_tests
```

## 1. Тестирование через GUI

Схема взаимодействия:

```text
GUI
 │
 │ gRPC: rs485_service.proto
 ▼
rs485_microservice
 │
 │ gRPC: rs485_driver.proto
 ▼
rs485_fake_driver
```

Перед запуском убедитесь, что порты из `config/config.yaml` не заняты другими процессами.

### Шаг 1. Запустить тестовый драйвер

В первом терминале:

```bash
./build-test/rs485_fake_driver
```

### Шаг 2. Запустить микросервис

Во втором терминале:

```bash
RS485_CONFIG_PATH="$PWD/config/config.yaml" \
./build-test/rs485_microservice
```

### Шаг 3. Запустить GUI

Соберите и запустите общий графический интерфейс в соответствии с инструкцией GUI-проекта.

Через GUI можно проверить:

* отправку данных методом `SendData`;
* запуск и остановку `Subscribe`;
* получение пакетов;
* обработку ошибок драйвера;
* поведение при отключении драйвера;
* завершение подписки при разрыве gRPC-потока.

## 2. Ручное тестирование без GUI

Схема взаимодействия:

```text
rs485_test_client
        │
        │ gRPC: rs485_service.proto
        ▼
rs485_microservice
        │
        │ gRPC: rs485_driver.proto
        ▼
rs485_fake_driver
```

### Шаг 1. Запустить тестовый драйвер

В первом терминале:

```bash
./build-test/rs485_fake_driver
```

### Шаг 2. Запустить микросервис

Во втором терминале:

```bash
RS485_CONFIG_PATH="$PWD/config/config.yaml" \
./build-test/rs485_microservice
```

### Шаг 3. Запустить тестовый клиент

В третьем терминале:

```bash
./build-test/rs485_test_client
```

После запуска выберите необходимый сценарий в консольном меню.

Этот вариант позволяет вручную проверить gRPC API микросервиса без участия GUI.

## 3. Автоматические интеграционные тесты

Автоматические тесты самостоятельно запускают внутри тестового процесса:

* `rs485_fake_driver` по адресу из секции `driver`;
* `Rs485ServiceImpl` по адресу из секции `service`;
* gRPC-клиент микросервиса.

Отдельно запускать `rs485_fake_driver` и `rs485_microservice` не нужно.

Перед запуском остановите вручную запущенные тестовый драйвер и микросервис, иначе их порты могут быть заняты.

Запуск всех автоматических тестов:

```bash
./scripts/run_tests.sh
```

При ошибке CTest выводит подробную информацию о завершившемся неудачно сценарии.

Тесты проверяют:

* успешную отправку данных через `SendData`;
* преобразование ошибки драйвера `NO_REPLY`;
* преобразование ошибки драйвера `TIMEOUT`;
* обработку внутренней gRPC-ошибки драйвера;
* отклонение запроса с пустыми данными;
* получение пакета через `Subscribe`.

Запустить GoogleTest напрямую можно командой:

```bash
./build-test/rs485_integration_tests
```

# Очистка тестовой сборки

Для удаления каталога `build-test` выполните:

```bash
./scripts/clean_tests.sh
```

Скрипт удаляет только результаты тестовой сборки и не затрагивает рабочий каталог `build`.

После очистки тестовую версию необходимо собрать заново:

```bash
./scripts/build_tests.sh
```

# Генерация gRPC-кода

C++- и gRPC-файлы для микросервиса генерируются автоматически во время рабочей или тестовой CMake-сборки:

```bash
./scripts/build.sh
```

или:

```bash
./scripts/build_tests.sh
```

Сгенерированные файлы микросервиса располагаются в каталогах:

```text
build/generated/
build-test/generated/
```

Они не должны редактироваться вручную.

Для GUI предусмотрен отдельный скрипт:

```bash
./scripts/generate_gui_proto.sh
```

Он генерирует клиентские файлы только для `rs485_service.proto` в каталог:

```text
build/gui_generated/
```

Эти файлы также не должны редактироваться вручную.
